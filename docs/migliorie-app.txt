Migliorie proposte per l'applicazione FAQ (web + mobile + backend + AWS)
Data: 2026-03-19

Obiettivo
- Rendere l'app piu robusta, semplice da usare e pronta per un uso reale in produzione.

Priorita alta (quick wins)
1) Login e gestione utenti
- Introdurre autenticazione (es. JWT o sessione) con ruoli minimi:
  - USER: compilazione e consultazione storico personale
  - ADMIN: funzioni amministrative
- Collegare ogni submission all'utente autenticato.
- Beneficio: sicurezza base, tracciabilita, separazione responsabilita.

2) Registrazione automatica indirizzi validi (sostituzione area Admin SES)
- Durante registrazione/login, validare l'email dell'utente con OTP/link di verifica.
- Mantenere un flag applicativo tipo email_verified=true/false.
- In fase di invio report, permettere invio solo a indirizzi verificati a livello applicativo.
- Se il provider e SES sandbox, usare lista mittenti/destinatari consentiti per evitare errori.
- Effetto: la sezione Admin manuale per verifica SES diventa superflua nel flusso utente.

3) UX errori e stato invio
- Messaggi errore uniformi tra web/mobile (rete, timeout, provider non attivo).
- Stato invio report piu ricco: IN_CODA, INVIATA, FALLITA, RITENTATIVO.
- Mostrare motivazione sintetica nel dettaglio submission.

4) Prevenzione doppi invii
- Aggiungere idempotenza su submit questionario.
- Disabilitare doppio click/tap e bloccare richieste duplicate lato backend.

5) Uniformita validazioni
- Stesse regole su web, mobile e backend per email, risposte, campi obbligatori.
- Ridurre mismatch e bug di comportamento tra canali.

Priorita media (stabilita e operativita)
6) Coda asincrona per email
- Separare persistenza questionario dall'invio email (es. SQS + worker).
- Retry con backoff e dead-letter queue per errori persistenti.

7) Osservabilita
- Correlation ID per richiesta end-to-end.
- Metriche (latenza, tasso errore, email inviate/fallite).
- Dashboard CloudWatch e allarmi base.

8) Hardening sicurezza
- Rate limiting su endpoint sensibili (submit/admin/auth).
- CORS restrittivo in produzione.
- Header sicurezza HTTP e HTTPS obbligatorio.

9) Gestione segreti
- Spostare credenziali in AWS Secrets Manager/SSM.
- Nessun segreto nei file locali condivisi.

10) Migliorie storico dati
- Filtri per data/email/esito invio.
- Paginazione backend e ordinamento consistente.
- Eventuale export CSV.

Priorita medio-alta (qualita software)
11) Test e quality gate
- Integration test con DB reale (Testcontainers).
- Test API contract.
- Test E2E principali percorsi utente.

12) CI/CD
- Pipeline automatica per test, lint, build docker, scan dipendenze.
- Deploy controllato con rollback semplice.

13) Evoluzione schema DB
- Indici su campi usati in ricerca/storico (es. submitted_at, patient_email).
- Strategia retention/archiviazione storica.

Priorita strategica (produzione)
14) Backup e disaster recovery
- Snapshot automatici RDS + restore testato periodicamente.
- Runbook operativo per incidenti principali.

15) Cost governance AWS
- Budget/allarmi costi.
- Tagging risorse e monitoraggio mensile.

16) Accessibilita e i18n
- Migliorie accessibilita (focus, contrasti, screen reader).
- Predisposizione multilingua (IT/EN) per UI e messaggi.

Roadmap suggerita
- Fase 1 (1-2 settimane): login, verifica email applicativa, dismissione area Admin SES, UX errori, idempotenza.
- Fase 2 (2-4 settimane): coda email, osservabilita, rate limiting, secrets manager.
- Fase 3 (4-8 settimane): CI/CD completa, test estesi, hardening produzione, backup/DR.

Nota importante
- La registrazione automatica degli indirizzi validi riduce il carico operativo e semplifica l'esperienza utente,
  ma lato AWS SES vanno comunque rispettate le regole dell'account (sandbox vs production access).
